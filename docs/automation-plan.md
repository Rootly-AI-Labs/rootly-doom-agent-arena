# Benchmark Batch-Run Plan

## Goal

Keep the current architecture intact:

- the browser starts the benchmark
- two external MCP agents stay outside the arena
- prompts and controller tokens are still generated by the existing flow

Add an alternative batch mode where one benchmark start runs `N` matches in sequence instead of one match at a time.

## What changes

Today the operator has to:

1. Start the arena.
2. Open two MCP agents.
3. Paste the two prompts.
4. Wait for the match to finish.
5. Click the next match manually.
6. Repeat until enough results exist.

The proposed batch mode keeps steps 1 to 3, then automates steps 4 to 6.

## Desired user flow

1. Open the arena UI.
2. Choose the usual matchup settings.
3. Set `Matches`, for example `20`.
4. Click `Start Benchmark`.
5. Paste the Player 1 prompt into one MCP agent and Player 2 prompt into the other.
6. Let the system auto-run all `20` matches.

The same two MCP agents stay connected for the whole batch.

## Current architecture we can reuse

Most of the batch structure already exists:

- the UI already has a `Rounds` control
- the server already tracks `duel_session_id`, `total_rounds`, and `current_round`
- `continue_session` already creates the next round in the same session
- controller tokens are already reused across rounds
- results already land under `session_*/round_*`

So this is mainly an auto-advance project on top of the current multi-round duel flow, not a new session model.

## Required codebase changes

### 1. UI: relabel and clarify batch mode

Reuse the existing round-count control, but present it as benchmark batch size.

Suggested shape:

- label: `Matches`
- default: `1`
- minimum: `1`
- button text: `Start Benchmark`

`1` preserves the current behavior.

## 2. Browser-driven auto-advance between matches

The browser should automatically drive the existing `continue_session` flow.

After a match reaches `finished`:

- wait briefly for the normal result artifacts to be written
- call `/api/arena/duel-session` with `continue_session=true`
- reload into the new `run_id`
- stop auto-advancing after the last match

This is the smallest implementation because today the next round already happens only when the browser posts a new duel-session request.

## 3. Stable agent contract across matches

The same two external MCP agents should be able to stay attached for the full batch.

That means:

- prompts must stop assuming the session ends after one match
- observations and match results must expose `run_id`, `current_round`, `total_rounds`, and `has_next_round`
- match boundaries must be explicit
- agents must know they need to call `set_participant_ready` again on each new round and reset `sequence_number`

This is the critical behavioral change. Without it, most agents will finish match 1 and stop.

## 4. Scheduling assumptions

For the first version, batch mode should repeat the current settings `N` times.

Do not claim seed rotation, side swapping, or scenario scheduling unless that logic is explicitly added. The current `continue_session` path increments the round number, but it does not build a matchup schedule.

## 5. Result layout

Keep the current per-match artifact layout under the existing session folder structure:

```text
benchmarks/results/session_*/
  round_01_run_*/
  round_02_run_*/
  round_03_run_*/
```

No new storage model is required.

## 6. Optional safety checks

Batch mode should stop cleanly if:

- one or both agents never send a first intent
- a match hangs past timeout
- the arena is reset manually mid-batch

Initial version can stop the batch and surface a clear error instead of trying to recover automatically. A dedicated stop/pause control can come later.

## Not required for this version

These are explicitly out of scope for the first batch-run implementation:

- headless MCP players
- replacing Codex or Claude with SDK-driven agents
- unattended model sweeps across many providers
- CI orchestration
- analysis pipeline rewrite

## Sequencing

| Phase | What | Outcome |
|---|---|---|
| 1 | Relabel the existing round control as `Matches` | UI reflects batch semantics |
| 2 | Auto-advance after `finished` through `continue_session` | Removes manual next-match clicking |
| 3 | Update prompts and MCP outputs for multi-match continuity | Same two agents can stay connected cleanly |
| 4 | Define first-version scheduling as repeat-current-settings | Avoids overstating benchmark coverage |
| 5 | Add stop/error handling | Batch fails loudly instead of drifting |

## Estimated effort

- UI relabeling: **less than half a day**
- browser-driven auto-advance: **1 day**
- prompt/session continuity cleanup: **half a day**
- failure handling polish: **half a day**

Total to first useful version: **~2 days**.
